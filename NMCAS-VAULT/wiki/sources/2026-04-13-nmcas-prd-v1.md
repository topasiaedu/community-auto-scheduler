# Source digest: `2026-04-13-nmcas-prd-v1.md`

**Raw path:** `raw/sources/2026-04-13-nmcas-prd-v1.md`

## One-line summary

Full V1 product requirements for NMCAS: a multi-project WhatsApp community scheduler supporting Post and Poll message types, built on Baileys + Supabase + pg-boss + Render/Vercel.

## Key claims (from source)

- **Purpose:** Eliminate manual copy-paste and real-time availability for community group posts. Send scheduled WhatsApp messages (Post or Poll) to WA groups.
- **Multi-project:** The top-level entity is a `Project`. Each project has one WA account, its own group list (fetched live), its own message queue, and its own notification recipient list.
- **WhatsApp:** Baileys (`@whiskeysockets/baileys`) — unofficial, personal WA, QR auth, WebSocket-based (no Puppeteer). Risk of account ban acknowledged; acceptable at low community volume.
- **Session storage:** Baileys auth state stored in **Supabase Storage** (private bucket, per-project folder) via custom adapter. Avoids Render Persistent Disk dependency.
- **Job queue:** `pg-boss` (Postgres-backed). No Redis. Uses existing Supabase Postgres.
- **Message types:** Post (text + optional image) and Poll (question, 2–12 options, single/multi-select). One-off only — no recurring.
- **Timezone:** MYT (UTC+8) hardcoded for V1. Stored as UTC in DB.
- **Failure notifications:** On `FAILED` status, WA message sent to project's `NotifyRecipient` list via that project's own Baileys instance.
- **Hosting:** Render Starter (always-on, required) + Vercel (FE) + Supabase (DB + Storage).

## Stated out-of-scope (V1)

Recurring messages, templates, multi-timezone, video type, individual recipients, full user accounts, WA Business API.
