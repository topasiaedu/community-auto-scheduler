## 1. Project overview

**NMCAS** (No More Copy-paste and Scheduling) is an internal web application that allows team members to compose, schedule, and auto-send WhatsApp messages to community groups. The app removes the need for manual copy-paste and real-time availability when posting to community groups.

The organisation runs **multiple projects**, each with its own WhatsApp account and community group(s). NMCAS must support all of them from a single hosted app.

**Target users:** Any team member, including interns. The UI must be learnable without documentation.

---

## 2. Core use case

1. A team member opens the app, selects a project, and picks a target group from that project's WhatsApp account.
2. They compose a **Post** (text + optional image) or a **Poll** (question + options).
3. They pick a date and time (Malaysia Time, UTC+8).
4. They hit Schedule.
5. At the exact scheduled time the system auto-sends the message via that project's connected WhatsApp number.
6. If the send fails, a WhatsApp notification is automatically sent to a configurable list of people.

---

## 3. Multi-project architecture

The top-level entity is a **Project**. Every resource in the system belongs to a project:

- Each project has exactly **one linked WhatsApp account** (phone number).
- Each project has its own **group list** (fetched live from that account).
- Each project has its own **scheduled message queue**.
- Each project has its own **notification recipient list** (who to alert on failure).

A user can switch between projects using a project selector in the app sidebar.

---

## 4. WhatsApp connectivity

- **Library:** Baileys (`@whiskeysockets/baileys`) — WebSocket-based, no Puppeteer, supports personal WA.
- **Auth flow:** QR code scan. Each project's WA session is authenticated by scanning a QR code shown in the app's Settings screen.
- **Session persistence:** Baileys session credentials (auth state JSON) are stored in **Supabase Storage** (private bucket, one folder per project). A custom Baileys `authState` adapter reads/writes from Supabase Storage instead of the local filesystem. This survives Render container redeploys without needing Render Persistent Disk.
- **Connection pool:** The API backend holds a pool of Baileys socket instances, one per project, keyed by `projectId`. Worker jobs look up the correct instance at send time.
- **Groups:** Fetched live using `groupFetchAllParticipating()` when the user opens the recipient picker. No caching layer.
- **Risk acknowledged:** Baileys is an unofficial WhatsApp client. Accounts are subject to WhatsApp ToS. Acceptable for internal, low-volume community use. A backup number should be kept per project for testing.

## 5. Message types (V1)

### 5.1 Post
- **Fields:** Target group (required), copy/text (required), image (optional)
- **WhatsApp behaviour:** Sends as a caption message if image is present; plain text otherwise
- **Image constraints:** Max 16 MB (WhatsApp limit); validated at upload time; stored in Supabase Storage (separate bucket from sessions)

### 5.2 Poll
- **Fields:** Target group (required), poll question (required), poll options (min 2, max 12 — WhatsApp native limit), single-select or multi-select toggle (user chooses per poll)
- **WhatsApp behaviour:** Sends as a native WhatsApp poll via Baileys `poll` message type
- **Note:** WhatsApp polls do not support a caption/intro text — the question string is the only text field

---

## 6. Scheduling

- All scheduling is **one-off** (no recurring/cron jobs in V1).
- Scheduled times are entered in **MYT (UTC+8)**, hardcoded for V1. Stored as UTC in the database.
- No minimum advance scheduling window (no "at least 15 min ahead" guard in V1).
- **Job queue:** `pg-boss` — a Postgres-backed job queue library. Uses the existing Supabase Postgres database. No Redis or separate queue infrastructure required.
- **Job flow:** On scheduling, a `pg-boss` job is created with a `startAfter` equal to the scheduled UTC datetime. At fire time, the worker sends via the correct Baileys instance and updates the message status.

---

## 7. Message statuses

| Status | Meaning |
|--------|---------|
| `PENDING` | Scheduled, not yet sent |
| `SENDING` | Job picked up by worker, send in progress |
| `SENT` | Successfully delivered to WhatsApp |
| `FAILED` | Send attempted but errored (see `error` field for reason) |

Failed messages trigger a WhatsApp notification to the project's notify list (see §9).

## 8. Data model

### Project
```
id           String   (cuid, PK)
name         String
description  String?
createdAt    DateTime
```

### ScheduledMessage
```
id              String      (cuid, PK)
projectId       String      (FK → Project)
groupJid        String      WA group JID (e.g. 1234567890@g.us)
groupName       String      Snapshot at scheduling time (display only)
type            Enum        POST | POLL
copyText        String?     Post body text (POST type)
imageUrl        String?     Supabase Storage URL (POST type, optional)
pollQuestion    String?     Poll question (POLL type)
pollOptions     String[]    Poll answer options (POLL type)
pollMultiSelect Boolean     Default false (POLL type)
scheduledAt     DateTime    UTC
status          Enum        PENDING | SENDING | SENT | FAILED
sentAt          DateTime?
error           String?     Last error message on failure
createdAt       DateTime
```

### NotifyRecipient
```
id          String   (cuid, PK)
projectId   String   (FK → Project)
waNumber    String   Full WA number e.g. 60123456789
label       String?  Display name e.g. "Stanley"
```

## 9. Failure notifications

- When a message transitions to `FAILED`, the system sends a WhatsApp message **to each `NotifyRecipient` for that project** using that project's own Baileys instance.
- Notification text: `"[NMCAS] Failed to send scheduled message to [groupName] at [scheduledAt MYT]. Error: [error]"`
- Configurable per project in Settings. Stored in `NotifyRecipient` table.
- Not configurable per message (global project setting only).

## 10. UI/UX

### Navigation (3 screens)
1. **Dashboard** — Upcoming and history message cards for the current project, with status badges.
2. **New Message** — Form with message type selector (Post / Poll), group picker (live), fields, date/time picker, Schedule button.
3. **Settings** — WhatsApp connection status + QR code panel, notify recipient list (add/remove).

### Project switcher
- Top-left of sidebar: dropdown of all projects with WA connection status dot.
- One "Add Project" entry at the bottom of the dropdown.
- Switching projects reloads dashboard data and groups for the new context.

### Key UX principles
- No modals for primary actions.
- Status badges colour-coded: yellow (PENDING), green (SENT), red (FAILED).
- Scheduled time displayed as: `"Sends in 3h 22m (Mon 14 Apr, 09:00 MYT)"`.
- Dashboard cards: one-click cancel (PENDING only), one-click duplicate.
- Group picker shows live list fetched from WA; no manual JID entry.
- Message type selector (Post / Poll) swaps form fields below it (segmented control).
- QR code in Settings auto-refreshes every 20 seconds when disconnected.
- Empty state on Dashboard is instructive: `"No messages yet — create your first one →"`.
- App is fully responsive; usable on mobile.

## 13. Out of scope (V1)

- Recurring/scheduled recurring messages
- Message templates
- Multi-timezone support (MYT only)
- Video message type
- Individual phone number recipients (groups only)
- Full user account system (single env-var password for now)
- Read receipts or delivery confirmation beyond WA API response
- WhatsApp Business API (personal WA only via Baileys)
